# Agent Orchestration Nurture — Email 1 of 3

**Subject line options:**

- **A (question):** Why does your coding agent go sideways?
- **B (declarative):** Your coding agent's failures aren't random.

---

If you've been using Copilot's coding agent for a while, you've seen both versions of it. The run that lands exactly where you pointed it. And the one that pursues something adjacent to what you asked — or wraps up cleanly and hands you output that falls apart the moment you look closely.

The easy read is that it's a prompting problem: clearer instructions would have prevented it. Sometimes that's true. But a lot of the inconsistency you're seeing follows a structure that has nothing to do with how well you wrote the spec. When an agent misreads your intent, it's usually because the instruction was specific enough to mean something — just not the same thing you meant. When it fabricates data instead of flagging an error, it's often because an integration point failed quietly and the agent filled the gap rather than surfacing the problem. Neither of those is random. Both are predictable once you understand the layer underneath.

The ebook covered the manager workflow: write a spec, assign the work, monitor the first five minutes to verify the agent understood what you asked, review the output. That workflow is real and it holds. But underneath it is an engineering layer that determines whether multi-agent work stays reliable at scale — or quietly compounds its own failure modes as you delegate more.

Here's where that layer gets explained: [Multi-agent workflows often fail. Here's how to engineer ones that don't.](https://github.blog/ai-and-ml/generative-ai/multi-agent-workflows-often-fail-heres-how-to-engineer-ones-that-dont/)
